最近有一個很明顯的體感：AI agent （諸如Codex或者Claude code）確實被講得越來越強，好像只要你願意交代任務，他就能幫你做完很多事情。寫程式、部署網站、整理資料、生成簡報、建立知識庫，甚至自動把文件變成網頁。

但我這幾天實際用下來，反而冒出一個有點反向的問題：

**AI 能做，不代表所有事情都該交給 AI 做。**

<!--more-->

這句話不是反 AI，也不是說 Codex 沒用。剛好相反，我是因為真的用了，真的看過他能做什麼，也真的感受到 token 額度被燒掉時那種「欸？我只是想做一件小事，怎麼成本這麼高？」的哭笑不得，才開始重新思考：在 AI agent 時代，我們需要學的也許不只是「怎樣使用 AI」，而是「怎樣判斷什麼時候不要使用 AI」。

這個思考的觸發點，其實從 6 月 30 日就開始了。

那時候我在看秦陽老師的 Codex 製作簡報教學。影片裡示範的是用 Codex 來做簡報，讓他處理檔案、生成內容、輸出成果。看起來確實很厲害，也很有未來感。但我一邊看，一邊心裡就有點懷疑：如果只是做一份簡報，真的一定要用 Codex 嗎？

因為我這陣子自己實測下來，很多簡報相關工作其實用 GPT 也能做。從討論主題、抓大綱、規劃頁面邏輯，到生成簡報圖片、再整理成 PDF 或網站用素材，這些事情在 GPT 對話裡就能完成。而且對我來說，簡報不只是「做出一個檔案」，而是要先想清楚：這份簡報到底要講什麼？它的主題線是什麼？圖片氣質要怎樣？文字要怎樣才像我？

這些不是單純的工程任務，而是創作與判斷。

所以當時我就已經有一個模糊的分工感：**如果只是單份簡報、單篇文章、單張圖片，GPT 其實已經很夠用；Codex 也許更適合處理一整批檔案、跨資料夾、需要本地專案結構的工作。**

只是那時候，這個想法還只是懷疑。直到這次，我真的用 Codex 去做一件看起來很適合交給 agent 的事情：更新網站、部署到 GitHub。

## Codex 很強，但真的會燒額度

事情是這樣的。

我原本只是想把一個做好的小工具連結放進教學網站。這個網站放在 GitHub 上，平常就是一些 HTML 小工具、頁面入口、連結整理。照理說，把一個檔案上傳、把入口加到首頁，應該不是什麼超困難的事情。

但因為它牽涉到 GitHub、網站部署、檔案路徑，我自己很習慣就覺得：「這好像是工程問題吧？那不如交給 Codex。」

結果就開始卡。

我只是給他一個網址，想讓他把它放到網站內。結果他一直失敗、一直嘗試、一直部署不成功。下午才做了一點事情，Codex 額度就用光；重置後又再試，結果又用光。那時候我真的有點傻眼：我到底做了什麼大工程？怎麼只是這麼簡單的更新網站就燒掉這麼多額度？

而且更麻煩的是，當 Codex 一直失敗，我會產生另一種焦慮：我不知道他到底在哪一步卡住。是檔案沒有上傳？是 GitHub Pages 沒有部署？是首頁沒有更新？還是我的檔案名稱有問題？

後來我才慢慢分辨出來：事情其實分成好幾層。

第一，檔案確實已經上傳到 GitHub repo 裡。  
第二，首頁工具卡片不會自動更新，所以就算檔案上傳了，首頁也不會自己冒出新入口。  
第三，GitHub Pages deployment 有時候會出現紅叉，甚至出現 build 成功、report 成功，但 deploy 失敗的狀況。  
第四，中文檔名雖然 GitHub 可以存，但在網址與部署上不一定好用，所以後來我把檔名改成英文。  
第五，首頁的卡片數量，例如「67 / 67 個工具」，也是手動寫在 `index.html` 裡，不會因為多了一個檔案就自動變成 68。

也就是說，這件事不是一個單純的「上傳失敗」。它其實混在一起：檔案管理、首頁目錄、GitHub Pages 部署、網址路徑、檔名規則。

一開始我不懂，所以覺得它是一大團黑箱。可是一旦拆開來看，它其實沒有那麼可怕。

## 很多事情不是難，只是我不知道路徑

後來我開始請 GPT 陪我一步一步看 GitHub。不是用 Codex 幫我改，而是先教我理解：

repository 是什麼？  
GitHub Pages 網址在哪裡？   
`content` 和 `static` 是做什麼的？  
哪些資料夾是網站內容更新的安全區？  
哪些像 `layouts`、`themes`、`.github/workflows` 這類資料夾，平常不要亂碰？

我原本連一個很基本的問題都會卡住：我按 GitHub 上的藍色連結，結果一直只進到 repo 設定頁面，那我要怎樣進到真正的網站網址？

聽起來好像很蠢，但其實這就是 GitHub 對初學者很不友善的地方。repo 是倉庫，網站是店面。你按到倉庫，不代表你看到店面。

後來我弄清楚自由人生實驗室的網站網址，知道 repo 名稱會對應到 `https://帳號.github.io/倉庫名稱/` 這樣的 GitHub Pages 網址，也知道有些 repo 比較特別，例如 `帳號.github.io` 本身就是使用者首頁 repo，所以網址直接是根目錄。

這些聽起來很技術，但真的理解之後，其實就是地圖。

而我真正學到的重點，不是「我會 GitHub 了」。我根本還不敢說自己會 GitHub。我只是學會了一個很小但非常有用的日常更新流程：

文章 `.md` 放到 `content` 裡對應的位置。  
圖片放到 `static/images`。  
Commit changes。  
等 GitHub Pages 自動部署。  
打開網站檢查首頁圖片與統計數據有沒有更新。

這件事我後來真的自己做了一次。

我上傳了一篇測試文章 `.md`，也上傳了圖片檔。一開始首頁的圖片沒有立刻出來，我還有點緊張，想說是不是又哪裡壞掉。結果過了一下，圖片正常顯示，統計數據也更新了。

整個流程花不到五分鐘，甚至可能連三分鐘都不到。

那一刻我真的有一種「啊？原來這樣就可以？」的感覺。

## 三分鐘上稿帶來的掌控感

這件事對我來說很重要。

因為它證明：有些事情不是很難，只是以前我不知道路徑。

GitHub 一開始看起來像工程師的黑魔法。Repo、commit、deployment、workflow、GitHub Pages、content、static……每一個字都讓人感覺自己是不是不該碰。可是實際拆開之後，日常上稿其實就是一條穩定的小路。

這種小路一旦走過一次，心裡會安很多。

我不需要懂完整的 Git。  
我不需要會命令列。  
我不需要理解整個 Hugo 主題怎樣運作。  
我也不需要每次都請 Codex 幫我部署。

但我至少知道：如果今天只是新增文章和圖片，我自己可以做。  
如果首頁圖片一開始沒出來，我可以先等一下，不用立刻慌。  
如果部署紅叉，我知道要去看 log。  
如果只是目錄沒有更新，我知道那可能不是檔案沒上傳，而是首頁沒有新增入口。

這就是掌控感。

不是那種「我變成技術高手」的掌控感，而是「我至少不再完全被黑箱嚇住」的掌控感。

## AI 能做，不代表我該交出去

這次經驗讓我最有感的，就是這句話：

**AI 能做，不代表我該交出去。**

現在大家很鼓吹 agent，好像只要一件事情能交給 AI，就應該交出去。彷彿使用 agent 越多，就代表越先進；越自動化，就代表越有效率。

但我這次的體驗剛好相反。

如果一件事情原本只需要我自己三分鐘就能完成，我卻交給 Codex，讓他讀一堆檔案、推理一堆路徑、嘗試部署、失敗重來，最後把額度燒光，那這真的是比較聰明的做法嗎？

我開始覺得，AI 時代真正重要的能力，不只是知道 AI agent能做什麼，而是能判斷：

這件事值得交給 AI agent嗎？  
這件事是不是我只要學會一次，以後就能自己快速完成？  
我把它交給 AI，是因為真的複雜，還是只是因為我害怕陌生介面？  
如果今天 token 額度用完，或 Codex 暫時不能用，我是不是就完全沒辦法更新自己的網站？

最後這個問題尤其刺到我。

如果自由人生實驗室是我的網站，是我用來承接生活、記錄、文章、影片與思考的地方，那我當然不希望它完全綁在某一個 agent 身上。

假如今天 Codex 不能用了，我難道就無法發文章？  
假如今天 token 額度不夠，我難道就不能更新圖片？  
假如今天 AI 部署失敗，我難道就只能乾等？

我不想這樣。

所以學會 GitHub 的基本上稿流程，不是為了證明我多厲害，而是為了替自己保留一點主權。

## 不是反 AI，而是重新分工

我並不是因此覺得 Codex 沒用。剛好相反，我覺得這次讓我更清楚每個工具該放在哪裡。

GPT 適合陪我想清楚。

像文章主旨、大綱、標題、簡報架構、封面風格、YouTube 說明欄、網站文章、自由人生實驗室的定位與語氣，這些都不是單純執行任務，而是需要來回討論、需要保留我的聲音與判斷。這些事情用 GPT 比較適合。

Codex 適合幫我搬重物。

如果有一天我要重構網站、修改很多檔案、跑本地專案、整理一大批資料、處理 GitHub Actions、建立自動化流程，那 Codex 很有價值。因為那是多檔案、多步驟、需要工程操作的工作。

但日常上稿，我自己做就好。

上傳文章、上傳圖片、確認路徑、commit、等部署、檢查首頁。這些事情一旦熟了，自己做反而最快，而且不消耗額度，也不會產生「我到底把事情交給誰了」的不安。

後來我給自己整理出一個很簡單的分工：

**GPT 陪我想清楚。  
Codex 幫我搬重物。  
GitHub 日常上稿我自己做。**

這句話目前很適合我。

## AI 很強，但也很花錢

這次還有一個很現實的問題，就是 token 和使用額度。

以前我可能不太會那麼具體感受到「AI 很花錢」。因為一般使用 GPT 對話時，比較像是訊息數限制。像我最近常用 GPT-5.5 High 聊比較深的內容，後來確認過，Plus 方案雖然也有限制，但那種限制對我的使用方式來說其實不容易爆。因為我通常打一大段，GPT 想很久回一大段，我再慢慢讀、慢慢回。這種深度對話，不太可能三小時傳出一百多則訊息。

但 Codex 的感覺完全不同。

Codex 做的不是單純聊天。他會讀檔、分析專案、修改、跑流程、嘗試部署、看錯誤，再重試。也就是說，就算我表面上只是下了一個指令，背後可能已經消耗了很多 token。

這也是為什麼我會覺得奇怪：我明明只是想做一件小事，怎麼下午額度就燒光？

所以我後來越來越覺得：AI 時代的使用成本，不只是「我有沒有訂閱」。還包括我是否知道什麼任務該用什麼工具，以及我有沒有能力避免把小任務交給昂貴的流程去處理。

這不只是省錢。  
這也是一種清醒。

## 外包不等於自由

這件事最後其實又繞回自由人生。

我一直在想自由：時間自由、物質自由、身體自由、精神自由、人際自由。很多時候談自由，很容易談到財務、工作、時間安排。但這次 GitHub 的小實驗讓我想到，自由有時候也包括一種很小的操作能力。

我能不能維護自己的網站？  
我能不能更新自己的文章？  
我能不能知道自己的資料放在哪裡？  
我能不能在工具失效時，仍然有基本的備援能力？  
我能不能選擇要自己做，還是交給 AI？

這些看起來都是小事，可是小事累積起來，其實就是一個人對自己生活系統的掌控感。

如果我每次遇到陌生介面就把它交給 agent，我表面上好像變得更有效率，但也可能慢慢失去對自己系統的觸覺。久了以後，我只知道「叫 AI 幫我修」，卻不知道檔案在哪、路徑在哪、錯誤在哪、自己能不能做。

那樣真的自由嗎？

我不確定。

至少對我來說，外包不等於自由。真正的自由比較像是：我可以選擇外包，也可以選擇自己做；我可以使用強大的工具，但不把自己的主權完全交出去。

## 工具應該讓人更自由，不是更依賴

最後我想把這次經驗收束成一句話：

**工具應該讓人更自由，不是讓人更依賴。**

AI agent 當然很強。Codex 也有他的價值。未來我還是會繼續用他，尤其是在需要處理多檔案、多步驟、自動化流程時。

但這次我學到的是，不能只被「AI 可以做到」這件事情所迷惑。還要問：

這件事交給 AI 之後，我有更自由嗎？  
我有更省力嗎？  
我有更清楚嗎？  
還是只是把原本三分鐘能完成的小事，變成一場昂貴又焦慮的黑箱任務？

有些事情，自己學會一次，反而更輕鬆。  
有些操作，保留在自己手上，反而更安心。  
有些小技能，不是為了成為專家，而是為了讓自己不被工具牽著走。

我想，這大概就是 AI 時代一種很小、但很重要的自由練習。

不是所有事情都該交給 AI agent。  
有些門，我還是想知道鑰匙在哪裡。  
因為這是我的網站。  
也是我的自由人生實驗室。
