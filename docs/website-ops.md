# 自由人生實驗室網站經營手冊

這份手冊的目的，是讓《自由人生實驗室》可以長期持續經營，而不是每次都靠意志力硬撐。

核心想法很簡單：

> 每一次更新，都要讓下一次更新更容易一點。

網站經營不是一次做完的大工程，而是一個可以一直玩下去的「無限遊戲」。重點不是追求完美，而是保留一個穩定、低摩擦、可以反覆回來的流程。

## 1. 這個網站目前怎麼運作

這是一個 Hugo 網站。

白話來說，Hugo 是一個「把文章檔變成網站」的工具。你平常主要改的是文章、影片頁、圖片和少量設定；真正把它變成網頁的工作，會由 Hugo 和 GitHub Pages 處理。

目前最重要的資料夾是：

- `content/posts/`：一般文章放這裡。
- `content/videos/`：YouTube 影片整理頁放這裡。
- `content/progress/`：實驗進度類內容放這裡。
- `static/images/`：文章封面、縮圖、網站圖片放這裡。
- `docs/`：網站工作手冊、流程規範、備忘錄放這裡。
- `scripts/`：自動化小工具放這裡。
- `layouts/`：網站畫面的模板放這裡，通常不需要常改。
- `hugo.toml`：網站基本設定放這裡，包含網站名稱、選單、主題設定。

給新手的安全建議：

- 最適合先改：`content/posts/`、`content/videos/`、`docs/`
- 需要比較小心：`hugo.toml`
- 不熟時先不要亂動：`layouts/`、`themes/`

## 2. 長期經營的基本節奏

不要把目標設成「把網站做好」。

這個目標太大，很容易累。

比較適合的節奏是：

- 每週至少留下一個公開痕跡。
- 每次只處理一種內容。
- 有草稿就先保留下來，不急著變成大文章。
- 有卡住的地方，就把卡住本身寫進手冊或交給 Codex 簡化。

一個很低壓的每週循環：

1. 週初：整理本週想留下的一個主題。
2. 週中：先產出粗稿，不急著完美。
3. 週末：挑一個最容易完成的版本發布。
4. 發布後：記錄這次哪個步驟最卡。
5. 下次：優先降低那個最卡的地方。

## 3. 內容可以分成三種

### 一般文章

適合比較完整的想法。

放在：

```text
content/posts/
```

適合的使用情境：

- 一篇完整反思
- 一個生活實驗
- 一段職涯或創作觀察
- 一個已經有明確主題的長文

可以直接對 Codex 說：

```text
幫我把這篇文章整理成網站文章並發布
```

### 影片頁

適合把 YouTube 影片也整理進網站，讓網站不是只靠文章成長。

放在：

```text
content/videos/
```

給 Codex 時，最少提供兩樣東西：

- YouTube 連結
- 想放在網站上的影片說明文字

可以直接對 Codex 說：

```text
幫我把這支 YouTube 影片建立成網站影片頁並發布
```

注意：影片標題要盡量使用 YouTube 原標題，不要自己重新改寫，避免網站與影片平台對不起來。

### 實驗進度

適合還沒成熟、但值得留下痕跡的東西。

放在：

```text
content/progress/
```

適合的使用情境：

- 一個還在試的流程
- 一個網站經營觀察
- 一個創作系統的進展
- 一個還不適合寫成正式文章的小紀錄

這類內容的價值是「保存過程」，不一定要像正式文章那麼完整。

## 4. 發布文章的最小流程

如果你只是想完成一次發文，最簡單的方式是把文章內容給 Codex，讓 Codex 代你處理。

你可以說：

```text
幫我發布這篇文章到自由人生實驗室
```

Codex 應該做的事：

1. 確認文章要放在 `content/posts/`。
2. 檢查文章開頭的基本資料。
3. 確認 `draft: false`，代表正式發布。
4. 確認圖片路徑正確。
5. 檢查文字格式有沒有明顯錯誤。
6. 提交並推送到 GitHub。
7. 告訴你網站接下來要去哪裡看。

如果要自己手動執行，目前有一個輔助工具：

```powershell
.\scripts\publish-new-content.ps1 -Message "publish new content"
```

這條指令的意思是：

- 找出新的文章、影片頁、圖片。
- 嘗試檢查網站能不能成功產生。
- 把這次變更送到 GitHub。

提醒：這條指令會發布內容，會讓網站開始更新。執行前要確認內容真的準備好了。

## 5. 發布影片頁的最小流程

新增影片頁時，先準備：

```text
YouTube 連結：
影片說明：
```

然後交給 Codex：

```text
幫我把這支影片新增到自由人生實驗室網站
```

Codex 應該做的事：

1. 從 YouTube 連結抓出影片 ID。
2. 優先確認 YouTube 原標題。
3. 在 `content/videos/` 新增影片文章。
4. 文章一開始放入 YouTube 嵌入碼。
5. 下方放你提供的影片說明。
6. 日期不要設定成未來時間。
7. 檢查後發布。

如果 YouTube 縮圖有問題，優先使用比較穩定的 `hqdefault.jpg`，不要直接假設 `maxresdefault.jpg` 一定存在。

## 6. 每週網站經營檢查清單

每週只要看這幾件事就好，不要把自己逼到太複雜。

```text
本週新增了什麼？
本週哪個想法值得留下？
有沒有影片可以整理成網站頁？
哪個操作最卡？
下週最小的一步是什麼？
```

適合每週對 Codex 說：

```text
幫我做一次自由人生實驗室網站經營盤點
```

Codex 可以協助：

- 看最近新增的文章和影片。
- 找出哪些內容還沒有整理成網站頁。
- 幫你列出下一個最小行動。
- 找出最常卡住的操作。
- 建議下一個可以自動化的小地方。

## 7. 卡住時先看這裡

### 網站發布後看不到新文章

先不要緊張，常見原因是：

- GitHub Pages 還在部署。
- 文章日期被設定成未來時間。
- 文章還是 `draft: true`。
- 檔案副檔名不是 `.md`。
- 瀏覽器還在顯示舊畫面。

可以請 Codex 說明並檢查：

```text
網站沒有看到剛剛發布的文章，幫我檢查原因
```

### 中文在終端機看起來亂碼

這通常不是文章壞掉，而是讀檔方式不適合中文。

比較安全的讀法是：

```powershell
Get-Content -Encoding UTF8 檔案路徑
```

這條指令的意思是：用適合中文的方式讀取檔案。

### Hugo 在本機不能跑

如果看到類似這句：

```text
Application Control policy has blocked this file
```

意思通常是 Windows 擋住本機 Hugo 執行，不一定是文章有問題。

這時候不要一直卡在本機預覽。比較務實的做法是：

- 檢查文章基本資料。
- 檢查文字格式。
- 推送到 GitHub。
- 讓 GitHub Actions 做正式建置。

GitHub Actions 可以想成：GitHub 那邊幫網站跑一次正式產生流程。

## 8. 建議保留的固定句型

你以後可以用這些短句啟動任務。

```text
幫我發布這篇文章
```

適合：文章已經準備好，只要放上網站。

```text
幫我把這支 YouTube 影片建立成網站影片頁
```

適合：有 YouTube 連結和影片說明。

```text
幫我檢查網站最近有哪些內容更新
```

適合：想知道網站目前長到哪裡。

```text
幫我找出這個網站經營流程最卡的地方
```

適合：覺得有點累，但不知道要先優化哪裡。

```text
幫我把這次操作整理進網站經營手冊
```

適合：發現一個新規則，想讓下次不要重踩。

## 9. 未來可以慢慢改善的地方

不要一次全做。每次只挑一個最能降低摩擦的項目。

優先順序建議：

1. 讓發布腳本在本機 Hugo 被 Windows 擋住時，不要直接中斷。
2. 建立文章範本，讓新文章不用每次重寫基本資料。
3. 建立影片頁範本，讓 YouTube 影片頁更快完成。
4. 建立每週盤點文件，記錄網站新增內容和卡點。
5. 建立草稿收集區，把未完成想法先存起來。
6. 建立「一鍵查看最近內容」的小工具。

## 10. 版本安全

Git 可以想成網站的「存檔系統」。

每次重要修改後，都應該留一個存檔點。這樣如果網站壞掉，才知道可以回到哪裡。

對新手來說，最重要的原則是：

- 一次只改一小塊。
- 發布前先確認這次改了哪些檔案。
- 不要把無關檔案一起發布。
- 大改網站畫面前，先提醒要留版本檢查點。

如果只是新增文章或影片頁，風險通常比較低。

如果要改 `layouts/` 或 `hugo.toml`，風險比較高，因為那會影響網站整體畫面或設定。

## 11. 新名詞白話解釋

- Hugo：把文章檔轉成網站的工具。
- Markdown：一種純文字文章格式，用簡單符號表示標題、清單、連結。
- Front matter：文章最前面的基本資料，例如標題、日期、分類、是否發布。
- draft：草稿。`draft: false` 代表正式發布，`draft: true` 代表先不要公開。
- Git：網站的存檔系統，可以記錄每次修改。
- commit：一次存檔點。
- push：把本機的修改送到 GitHub。
- GitHub Pages：GitHub 提供的網站發布服務。
- GitHub Actions：GitHub 上自動幫你跑流程的工具，例如產生網站。
- 部署：把檔案變成真正看得到的網站。
- 腳本：幫你自動做重複工作的指令小工具。

## 12. 這份手冊的使用方式

這份手冊不是一次寫完就封存。

每次發現新的卡點，就把它補進來。

每次把一個操作變簡單，就把新的做法補進來。

如果網站經營是一個無限遊戲，這份文件就是遊戲規則和存檔說明。
